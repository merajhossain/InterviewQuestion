1. How do you handle conflict in a team?

Senior/Lead
Simple Explanation
Listen to all sides, focus on the problem (not people), and find a solution that aligns with team goals.
Detailed Answer

Steps for conflict resolution:

  1. Listen actively: Understand each perspective without judgment
  2. Identify the root cause: Is it technical disagreement, miscommunication, or resource conflict?
  3. Focus on data: Use metrics, user feedback, or technical constraints to guide decisions
  4. Find common ground: What does everyone agree on?
  5. Propose solutions: Offer multiple options and discuss trade-offs
  6. Follow up: Ensure the resolution is working

2. Describe a time you mentored a junior developer.

Senior/Lead
Simple Explanation
Share a specific example using the STAR method (Situation, Task, Action, Result).
Detailed Answer

Effective mentoring includes:

  • Pair programming: Work together on complex features
  • Code reviews: Provide constructive feedback with explanations
  • Resource sharing: Recommend articles, courses, and documentation
  • Goal setting: Help them define learning objectives
  • Safe environment: Encourage questions and experimentation
// Code Review Example - Teaching Moment
// Instead of: "This is wrong"
// Say: "Great start! Here's how we can improve this..."

// Before (Junior's code)
const users = data.map(u => u.name);

// Feedback with explanation:
// "Consider using destructuring for cleaner code:
const users = data.map(({ name }) => name);

// Also, what happens if 'name' is undefined?
// Let's add a fallback:
const users = data.map(({ name = 'Unknown' }) => name);"

3. How do you prioritize technical debt vs new features?

Senior/Lead
Simple Explanation
Balance business needs with code health. Use the "20% rule" - dedicate ~20% of sprint capacity to tech debt.
Detailed Answer

Framework for prioritization:

  • Impact vs Effort Matrix: Prioritize high-impact, low-effort items
  • Risk Assessment: Critical bugs and security issues come first
  • Developer Velocity: If tech debt slows down feature development, address it
  • Communicate trade-offs: Help stakeholders understand the cost of ignoring tech debt
  • Incremental approach: Don't wait for a "big rewrite"

4. Tell me about a production incident you handled.

Senior/Lead
Simple Explanation
Describe the incident, your response, resolution, and lessons learned. Focus on process and communication.
Detailed Answer

Incident response framework:

  1. Detect: Monitoring alerts, user reports
  2. Assess: Severity, impact, affected users
  3. Communicate: Notify stakeholders, create status page
  4. Mitigate: Quick fix or rollback
  5. Resolve: Root cause analysis and permanent fix
  6. Post-mortem: Document and prevent recurrence

5. How do you foster a culture of code quality?

Senior/Lead
Simple Explanation
Lead by example, establish standards, automate checks, and make code reviews a learning opportunity.
Detailed Answer

Building quality culture:

  • Automated standards: ESLint, Prettier, TypeScript strict mode
  • Code review guidelines: Clear, constructive, timely
  • Testing requirements: Minimum coverage thresholds
  • Documentation: README, ADRs (Architecture Decision Records)
  • Knowledge sharing: Tech talks, pair programming sessions
  • Celebrate quality: Recognize well-written code

6. How do you handle estimation and deadlines?

Senior/Lead
Simple Explanation
Break down tasks, add buffer for unknowns, communicate risks early, and push back on unrealistic deadlines.
Detailed Answer

Estimation best practices:

  • Story points: Use relative sizing (Fibonacci: 1, 2, 3, 5, 8, 13)
  • Three-point estimation: Best case, likely case, worst case
  • Include everything: Testing, code review, deployment, documentation
  • Track velocity: Use historical data to improve estimates
  • Communicate uncertainty: "This is a rough estimate with ±30% variance"
  • Negotiate scope: If deadline is fixed, what can we cut?
// Estimation Example
const feature = {
  name: "User Dashboard Redesign",
  breakdown: [
    { task: "Design review & feedback", points: 2 },
    { task: "Component architecture", points: 3 },
    { task: "Implement UI components", points: 8 },
    { task: "API integration", points: 5 },
    { task: "Unit tests", points: 3 },
    { task: "E2E tests", points: 5 },
    { task: "Code review iterations", points: 2 },
    { task: "Documentation", points: 1 }
  ],
  total: 29, // points
  buffer: 6, // 20% buffer for unknowns
  estimated_total: 35, // points
  velocity: 20, // points per sprint
  estimated_sprints: 2 // Math.ceil(35/20)
};

7. Describe a time you disagreed with a decision.

Senior/Lead
Simple Explanation
Show you can disagree constructively, present alternatives with data, and commit once a decision is made.
Detailed Answer

Principles of constructive dissent:

  • Disagree and commit: Voice concerns, but support the final decision
  • Data over opinions: Back your position with evidence
  • Propose alternatives: Don't just say "no", offer solutions
  • Consider context: Understand constraints you might not be aware of
  • Escalate appropriately: Know when to involve higher-level decision makers
  • Learn from outcomes: Were your concerns valid? What can you learn?

8. How do you evaluate candidates during interviews?

Senior/Lead
Simple Explanation
Look for problem-solving ability, communication skills, cultural fit, and growth potential - not just technical knowledge.
Detailed Answer

Evaluation framework:

  • Technical competency: Can they solve problems? Do they write clean code?
  • Communication: Can they explain their thinking clearly?
  • Collaboration: How do they handle feedback? Do they ask good questions?
  • Learning ability: How do they approach unfamiliar problems?
  • Cultural fit: Do they align with team values?
  • Avoid bias: Use structured interviews, diverse interview panels

10. Tell me about a time you failed and what you learned.

Senior/Lead
Simple Explanation
Show vulnerability, accountability, and growth. Focus on the learning and how you've applied it since.
Detailed Answer

Framework for discussing failure:

  • Own it: Take responsibility, don't blame others
  • Context: What led to the failure?
  • Impact: What were the consequences?
  • Response: How did you handle it?
  • Learning: What did you take away?
  • Application: How have you applied this learning since?
  • Growth mindset: Failures are learning opportunities
// Example Failure Story Structure
const failureStory = {
